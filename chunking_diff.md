# Chunking Strategy: v1 vs v2

## What Changed
In v1 (the earlier BM25 prototype), chunking was likely naive—perhaps splitting on a fixed character count or relying solely on double newlines to segment the text. In v2, the chunking pipeline is highly structural and content-aware. First, it uses `PyMuPDF` to parse the specific page ranges for the requested chapters, completely bypassing irrelevant index pages. Second, it uses a tiered splitting approach: it splits by paragraphs, classifies the content type (`prose`, `worked_example`, `question_or_exercise`), and merges small prose blocks up to a strict `tiktoken` limit of 250 tokens. It employs a fallback sentence-boundary split (`(?<=[.!?])\n`) to break up dense pages that lack standard paragraph spacing. Finally, it ensures that distinct semantic units, such as numerical exercises and example problems, are kept in their own isolated chunks and tagged with appropriate metadata.

## What Improved
This context-aware strategy prevents "franken-chunks" where a question and a completely unrelated prose paragraph are merged into the same block, leading to hallucinated context. Setting a hard 250-token limit dramatically improves retrieval density, allowing the cosine similarity to strongly correlate with actual answer presence rather than matching on long, diluting passages. Tagging the chunks with a `content_type` enabled our Stage 5 fix, where we applied a runtime metadata filter to penalize `question_or_exercise` chunks. This successfully prevented the retriever from favoring keyword-dense questions over the actual factual definitions, rescuing questions like "What is the value of G?" from false refusals.
